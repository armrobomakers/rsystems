# FRONTEND MIGRATION OPTIONS

Дата проверки: `2026-07-12`

## Исходные условия

Сейчас сайт `R Systems` - статический лендинг без React/Next/Tailwind/shadcn.
Главная цель - улучшить интерфейс и повысить конверсию, а не провести инфраструктурный рефакторинг ради самого рефакторинга.

## Вариант A - без миграции

### Суть

- оставить текущий стек: HTML + CSS + vanilla JS
- применять `UI UX Pro Max` для анализа и визуального направления
- использовать `21st.dev` как источник референсов и готовых паттернов
- вручную адаптировать найденные решения под текущую верстку

### Преимущества

- минимальный риск
- почти нет шансов сломать текущий сайт
- не нужно трогать backend-часть
- не нужно менять URL-структуру
- не нужно перестраивать деплой
- можно быстро выпускать улучшения

### Недостатки

- нельзя напрямую вставлять `21st.dev` React/shadcn-компоненты
- часть готовых решений придется пересобирать руками
- сложные интерактивные паттерны делать неудобнее, чем в React

### Риски

- визуальная реализация может быть чуть менее “из коробки”, чем в React/shadcn
- есть риск ограничиться только косметикой, если не держать четкий дизайн-бриф

### Объем изменений

- низкий
- в основном: текущие HTML/CSS/JS файлы

### Какие файлы придется менять

- [`index.html`](./index.html)
- [`styles.css`](./styles.css)
- [`app.js`](./app.js)
- возможно [`header-motion.html`](./header-motion.html) как отдельный прототип, если захочется вынести motion-эксперименты

### Какие зависимости появятся

- никаких новых обязательных зависимостей

### Влияние на backend

- нулевое
- Telegram CLI и скрипты остаются как есть

### Влияние на URL и SEO

- URL можно сохранить
- якорная навигация сохраняется
- SEO-риски минимальны, потому что структура не меняется

### Влияние на авторизацию

- никакого

### Влияние на деплой

- без изменений
- Vercel и Docker продолжают работать как статический хостинг

### Вероятность нарушения текущей функциональности

- очень низкая

## Вариант B - частичная модернизация

### Суть

- сохранить текущую архитектуру
- добавить только совместимые инструменты точечно
- при необходимости использовать “острова” интерактива или небольшие модули
- не переписывать весь сайт

### Что это может означать на практике

- оставить основной лендинг статическим
- вынести тяжелую интерактивность в отдельные самодостаточные блоки
- использовать JS-модули или маленький отдельный фронтенд для конкретной задачи

### Преимущества

- меньше рисков, чем при полной миграции
- можно улучшить только нужные части
- можно постепенно подготовить путь к более современному стеку

### Недостатки

- сложнее архитектура, чем у варианта A
- часть логики станет гибридной
- командная поддержка может усложниться

### Риски

- можно получить “полумиграцию”, где стек сложнее, а ценность еще не максимальна
- интеграция интерактивных блоков может привести к техническому зоопарку

### Объем изменений

- средний
- больше, чем у A, но меньше, чем у C

### Какие файлы придется менять

- [`index.html`](./index.html)
- [`styles.css`](./styles.css)
- [`app.js`](./app.js)
- возможно появятся новые JS-модули или сборочные конфиги

### Какие зависимости появятся

- зависит от реализации
- если делать действительно гибридно, могут появиться сборщик, JSX/TS toolchain или легкие UI-библиотеки

### Влияние на backend

- обычно минимальное
- если появятся формы/лиды, может потребоваться серверный endpoint или serverless-функция

### Влияние на URL и SEO

- при аккуратной реализации можно сохранить текущую схему
- если появятся новые блоки/страницы, потребуется следить за метаданными и редиректами

### Влияние на авторизацию

- пока не требуется

### Влияние на деплой

- может потребоваться адаптация Vercel-конфига или сборки
- Docker-сценарий тоже может потребовать обновления

### Вероятность нарушения текущей функциональности

- средняя

## Вариант C - полноценная миграция frontend

### Суть

- переписать фронтенд на React или Next.js
- добавить TypeScript
- подключить Tailwind CSS
- добавить shadcn/ui
- создать `components.json`
- строить UI с прицелом на прямое использование 21st.dev

### Преимущества

- максимальная совместимость с `21st.dev`
- удобнее масштабировать UI
- легче собирать переиспользуемые компоненты
- проще развивать сайт в продуктовый интерфейс, а не только в лендинг

### Недостатки

- самый дорогой и рискованный вариант
- текущий лендинг придется пересобирать
- есть риск потерять скорость и точность текущей версии
- потребуется больше инфраструктуры и дисциплины

### Риски

- ломается текущее поведение шапки, анимаций и адаптива
- возможны проблемы с SEO при неправильной миграции
- возможна потеря скорости поставки результата
- можно перегрузить проект стеком, который сейчас не нужен

### Объем изменений

- высокий
- фактически это новый frontend-проект поверх существующего репозитория

### Какие файлы придется менять

Вероятно почти все frontend-артефакты, включая:

- [`index.html`](./index.html) -> станет точкой входа или будет заменен
- [`styles.css`](./styles.css) -> перенесется в Tailwind / tokens / component styles
- [`app.js`](./app.js) -> частично или полностью исчезнет
- `package.json` -> новые скрипты, зависимости, build tooling
- `vercel.json` -> новые rules под framework output
- `Dockerfile` / `docker-compose.yml` -> под новый build/runtime
- новые файлы вроде:
  - `app/layout.tsx`
  - `app/page.tsx`
  - `components.json`
  - `tailwind.config.*`
  - `postcss.config.*`
  - `tsconfig.json`

### Какие зависимости появятся

- React
- React DOM
- Next.js или другой React framework
- TypeScript
- Tailwind CSS
- shadcn/ui
- likely `components.json`-driven setup
- дополнительные инструменты для UI библиотеки и линтинга

### Влияние на backend

- напрямую минимальное, потому что Telegram-логика живет отдельно
- но если появятся формы, лидогенерация или кабинет, понадобятся новые API routes / serverless endpoints

### Влияние на URL и SEO

- изменится структура роутинга
- потребуется продумать редиректы и canonical URLs
- если сделать неправильно, можно потерять часть SEO-стоимости текущего лендинга

### Влияние на авторизацию

- на лендинге сейчас авторизация не нужна
- в React/Next это может появиться позже, если сайт превратится в продукт

### Влияние на деплой

- потребуется новый pipeline
- нужно будет убедиться, что Vercel и Docker знают, как собирать новый frontend

### Вероятность нарушения текущей функциональности

- высокая

## Сравнение

| Параметр | A - без миграции | B - частичная модернизация | C - полная миграция |
|---|---:|---:|---:|
| Риск | низкий | средний | высокий |
| Скорость | высокая | средняя | низкая |
| Совместимость с 21st.dev | низкая, только как референс | средняя | высокая |
| Стоимость изменений | низкая | средняя | высокая |
| Вероятность сломать текущий сайт | низкая | средняя | высокая |
| Нужна ли новая инфраструктура | нет | возможно | да |
| Подходит для текущего лендинга | да | да, если есть точечная интерактивность | скорее нет, если цель только маркетинг |

## Итоговая оценка

Для текущего проекта лучший баланс дает **вариант A**.

Почему:

- сайт уже работает как статический лендинг
- текущая цель - быстро улучшить конверсию и визуальное качество
- `21st.dev` не дает прямой выгоды без переписывания фронтенда
- полный React/Next-переход сейчас добавит больше риска, чем пользы
