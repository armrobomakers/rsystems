# RECOMMENDED UI PATH

Дата проверки: `2026-07-12`

## Рекомендация

**Выбираю вариант A - без миграции.**

Это минимально рискованный путь, который позволяет:

- улучшить интерфейс уже сейчас
- использовать `UI UX Pro Max` для дизайн-аналитики
- использовать `21st.dev` как источник сильных референсов и паттернов
- не ломать текущий статический сайт
- не переписывать проект ради React/shadcn, которые пока не окупаются

## Почему именно A

### 1. Текущий сайт слишком простой для полной миграции

Сейчас это один лендинг с якорями, без форм, без авторизации, без API и без сложной продуктовой логики.

Для такой поверхности:

- React/Next не дают достаточного выигрыша, чтобы оправдать миграцию
- Tailwind/shadcn нужны только если есть реальная потребность в переиспользуемых компонентах и сложной системе

### 2. `21st.dev` пока полезнее как референс, чем как зависимость

Поскольку в проекте нет React/Next/Tailwind/shadcn/components.json, прямое использование 21st.dev потребует большого рефакторинга.

Это не соответствует текущей задаче:

- быстро поднять качество сайта
- не потерять текущую работоспособность
- не усложнять стек без необходимости

### 3. Минимальный риск сейчас важнее идеальной совместимости

Текущий сайт уже выполняет главную функцию:

- показывает оффер
- ведет в Telegram
- выглядит как продающий лендинг

Значит, выгоднее:

- довести контент
- улучшить визуальный ритм
- усилить шапку, hero, доверие и CTA
- собрать кейсы и микро-доказательства

чем строить новый frontend-процесс.

## Как использовать skills без миграции

### `UI UX Pro Max`

Использовать для:

- выбора визуального направления
- проверки премиальности интерфейса
- подбора композиции, типографики, плотности и контраста
- оценки, где лендинг выглядит “шаблонно”, а где “дорого”

### `21st.dev`

Использовать только как:

- базу референсов
- каталог сильных UI-паттернов
- источник идей для hero, nav, cards, stats, sections

Не использовать как прямую зависимость, пока стек не мигрирован.

## Когда имеет смысл перейти к варианту C

Полную миграцию имеет смысл делать только если появится хотя бы несколько из этих условий:

- на сайте станет 3+ самостоятельных страниц
- появится кабинет, форма, лид-воркфлоу или сложная воронка
- появится большая библиотека переиспользуемых UI-компонентов
- нужно будет активно брать и адаптировать `21st.dev` components
- появится backend-интеракция прямо на сайте

Если этого нет, React/Next - лишняя стоимость.

## Практический план после этой проверки

1. Зафиксировать текущий статический стек как базу.
2. Использовать `UI UX Pro Max` для точечной UI-оценки.
3. Использовать `21st.dev` только для референсов и component inspiration.
4. Доработать текущий лендинг вручную в HTML/CSS/vanilla JS.
5. Пересмотреть миграцию только после появления реальной продуктовой нагрузки.

## Итог

Сейчас самый правильный путь - **не мигрировать**.

Это дешевле, быстрее и безопаснее, а для текущей задачи дает почти весь практический выигрыш без архитектурного долга.
